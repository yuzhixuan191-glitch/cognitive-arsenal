# 构建日志：从 Excel 创作台到个人知识库网站

> 完整记录这个项目从一个 Excel 文件变成一个在线知识库的全过程。包括每个阶段的决策思路、遇到的问题、以及解决方案。

---

## 第零章：起点——一个 Excel 文件

### 输入

一个名为「2025年-创作台-12月.xlsx」的 Excel 文件。打开后发现：

- **21 个 sheet 页**，每个 sheet 以日期或主题命名（如"巴别塔"、"9-蓝战非"、"01.27-写作协议栈v1.0-v5.1"）
- **没有统一的表格结构**——文章不是按行排列的，而是散落在不同单元格里（B1、D2、E3、F4……）
- 每个单元格的内容长度从几十字到几万字不等
- 内容是 Markdown 格式的中文文章，包含标题、粗体、引用、代码块、列表等

这不是一份"数据表"，更像是一个**创作者的工作台**——文章在不同列里展开，像是多条并行的思维轨道。

### 第一个判断

传统的"读取 Excel → 按行解析"思路在这里行不通。我需要：
1. 遍历每个 sheet 的每一个单元格
2. 过滤掉空值和过短的内容（<50 字）
3. 把每个有实质内容的单元格当作一篇独立文章

---

## 第一章：数据探索与提取

### Step 1: 探测结构

用 `openpyxl` 打开文件，逐个 sheet 扫描所有单元格。关键发现：

```
巴别塔: 16 篇实质内容
9-蓝战非: 4 篇
10-索亚: 43 篇（内容最密集的 sheet 之一）
...
01.30-混沌文档→剧本式写作: 60 篇
02.25-游泳健身复盘→ACAL: 48 篇
```

**总计 735 篇文章，横跨 21 个 sheet。**

### Step 2: 标题提取

文章没有专门的"标题"字段，标题藏在内容里。我写了一个优先级提取策略：

1. 第一优先：Markdown 标题（`# 标题` 或 `## 标题`）
2. 第二优先：书名号标题（`《标题》`）
3. 第三优先：加粗文本（`**标题**`）
4. 兜底：取第一个非空行的前 80 个字符

大多数文章能正确提取标题，但有一些边缘情况——比如以"好的，收到指令。"开头的 AI 回复，或者以"deepseek大神你好"开头的对话投喂。这些后来在体裁分类阶段得到了处理。

### Step 3: 生成预览

取每篇文章的前 150 个字符作为预览文本，去掉换行符。这个预览文本会显示在文章卡片上，也用于搜索。

---

## 第二章：V1——单文件网站（首次尝试）

### 设计决策

最初的方案很直接：把所有数据嵌入一个 HTML 文件里。

- 左侧导航栏列出 21 个原始分类
- 右侧显示文章卡片
- 点击打开侧滑阅读面板
- Markdown 用 `marked.js` 渲染

### 视觉方向

内容的主题是"认知军火库"——用户自称"Refine无羡"，文章涉及认知工程、AI 协作、写作系统。我选择了：

- **深色工业风**：黑色底 (#08080a) + 黄铜金 (#c9a84c)
- **字体组合**：Noto Serif SC（标题）+ Noto Sans SC（正文）+ JetBrains Mono（元数据）
- **质感**：SVG 噪点叠层模拟胶片颗粒感

### 结果

单个 HTML 文件 **10.3MB**。功能完整，本地打开没问题。但这个体积为后面的部署埋下了隐患。

## 第三章：V2——智能分类系统

### 用户的新需求

> "有些内容其实是属于纪要，有些是属于文章，有些是剧本……我希望你有你自己的想法，让知识库整理得更好一些"

这意味着原始的 21 个 sheet 分类不够——它们是按时间/项目组织的，但同一个 sheet 里可能混着原创文章、AI 纪要、Prompt 协议等完全不同体裁的内容。

### 体裁分类算法

我设计了一个基于关键词的规则分类器（而非机器学习，因为这里规则足够清晰）：

```
优先级从高到低：
1. 对话/投喂 → "deepseek大神"、"投喂"、"压力测试模式"
2. 系列规划 → "系列总标题"、"总纲"、"完整架构"（且不含"篇原创"）
3. 剧本/蓝图 → "剧本"、"创作蓝图"（且不含"篇原创"）
4. 认知纪要 → "纪要"、"认知演进"、"撰文者注"、"认知考古学家"
5. 原创文章 → "Refine无羡的第"、"篇原创"、"简短摘要"、"沉浸时间"
6. Prompt/协议 → "协议"、"Prompt"、"协处理器"、"启动指令"
7. 复盘/归档 → "复盘"、"归档"、"ACAL"、"EventChain"
8. 兜底 → 其他/杂记
```

**关键设计决策**：优先级顺序很重要。"对话/投喂"必须排在最前面，因为用户的原始对话里可能提到"文章"、"纪要"等词汇，但它们本身不是那些体裁。同理，"篇原创"是原创文章的强信号，所以剧本和系列规划的判断里要排除包含这个关键词的内容。

### 主题分类

主题分类更简单，主要基于领域关键词 + sheet 名称兜底：

```
巴别塔·悬搁哲学 ← "巴别塔"、"悬搁"
艾梵·认知协处理器 ← "艾梵"、"Aevum"、"协处理器"
健身·肉身道场 ← "健身"、"游泳"、"教练"、"体脂"
...
通识·认知探索 ← 兜底分类
```

### 最终分类结果

| 体裁 | 数量 |
|------|------|
| 原创文章 | 361 |
| 认知纪要 | 114 |
| Prompt/协议 | 91 |
| 其他/杂记 | 91 |
| 剧本/蓝图 | 34 |
| 对话/投喂 | 21 |
| 系列规划 | 17 |
| 复盘/归档 | 6 |

### 三维导航

最终设计了三种浏览维度：
- **体裁分类**（文章是什么类型）
- **主题专栏**（文章讲什么内容）
- **创作时间线**（文章何时创作，保留原始 sheet 结构）

这三者不是互斥的。同一篇文章属于一个体裁、一个主题、一个时间线分组。用户可以从任意维度进入。

### 首页仪表盘

加了 6 张 ECharts 图表，让用户一眼看到整个知识库的全貌：体裁分布、主题分布、字数分布、时间线产出、体裁×主题热力图、各主题累计字数。

## 第四章：部署踩坑——10MB 的教训

### GitHub Pages 部署

用 `gh` CLI 创建了私有仓库，推送代码，启用 Pages。

**第一个坑**：GitHub 免费版不支持私有仓库的 Pages。解决方案：改为公开仓库。

**第二个坑（核心问题）**：10MB 的 HTML 文件推到 GitHub 后，GitHub 无法在网页上预览（"Sorry about that, but we can't show files that are this big right now"）。虽然 Pages 理论上能 serve 这个文件，但加载体验极差。

### 数据分离方案

将单一巨型文件拆分为：

```
index.html          → 52KB（应用壳 + CSS + JS）
data/index.json     → 427KB（文章索引：标题、分类、摘要，无正文）
data/manifest.json  → 映射表（origin → chunk 文件名）
data/chunk_0.json   → 巴别塔的 16 篇文章正文
data/chunk_1.json   → 蓝战非的 4 篇文章正文
...共 21 个 chunk
```

**加载策略**：
1. 启动时加载 `index.json`（427KB）和 `manifest.json`——用户立即能看到所有文章标题和分类
2. 用户点击某篇文章时，按 `origin` 字段查找对应的 chunk 文件，`fetch` 加载
3. 加载一个 chunk 后，该 chunk 内所有文章的正文都被缓存——同一时间线分组内的切换是即时的

**效果**：首屏加载从 10MB 降到 ~500KB，GitHub 可以正常预览代码，Pages 加载体验也好了很多。

### 代码改造

这次重构涉及的关键代码变更：
- `const ARTICLES = [嵌入数据]` → `async function loadData()` + `fetch`
- `openArt()` 从同步变为 `async`，打开文章时先显示"加载中..."
- 搜索从全文搜索降级为标题+摘要+分类搜索（因为正文不在内存里）

搜索能力的降级是一个有意识的取舍。如果未来需要全文搜索，可以构建一个倒排索引文件，但目前标题+摘要搜索已经覆盖了绝大多数使用场景。

## 第五章：文章工具栏

### 用户需求

> "搜索到一篇文章，点开阅读，上方有一个一键复制、一键分享到微信/QQ/抖音、一键为这篇文章转化为图片的功能（类似 flomo 笔记那种）"

### 一键复制

调用 `navigator.clipboard.writeText()`，带 fallback（创建临时 textarea + `execCommand('copy')`）。复制内容是"标题 + 换行 + 全文"。

### 分享菜单

中国互联网的分享生态和国际不一样。没有统一的 Web Share API 可用，各平台的情况：

| 平台 | 方案 |
|------|------|
| 微信 | 无网页端分享接口，只能复制内容+提示用户去微信粘贴 |
| QQ | `connect.qq.com/widget/shareqq` 分享弹窗 |
| 微博 | `service.weibo.com/share/share.php` 分享页面 |
| 抖音 | 无网页端分享接口，复制链接+提示 |
| 复制链接 | 生成带文章 ID 的锚点链接 |

### flomo 风格卡片图

这是最有趣的功能。设计了一张独立于网站主题的卡片：

- **暖白底色** (#fffdf7) 而非网站的深色——因为卡片是用来分享的，浅色在社交媒体上更醒目
- **金色顶部条**：渐变色带，品牌识别
- **衬线标题**：Noto Serif SC，突出文章标题
- **正文摘要**：取纯文本前 400 字，底部用渐变淡出暗示"还有更多"
- **标签栏**：体裁 + 主题，浅色药丸标签
- **底部来源**：等宽字体显示原始分类

用 `html2canvas` 将 DOM 节点渲染为 canvas，再导出 2x 分辨率的 PNG。这里有个细节：卡片的 CSS 使用了固定颜色值而非 CSS 变量，因为 `html2canvas` 在某些情况下对 CSS 变量的支持不完美。

## 第六章：回顾与反思

### 关于"AI 自动分类"的诚实评估

基于关键词的分类器准确率大约在 85-90%。它在以下情况会犯错：

- 一篇原创文章如果正文里引用了"纪要"这个词，可能被误分为认知纪要
- 一些短小的杂记如果恰好提到了"协议"，可能被归为 Prompt/协议
- "其他/杂记"成了一个较大的兜底类别（91 篇），其中一部分可能有更精确的归属

如果要提升准确率，可以：
1. 用 LLM 对每篇文章做一次性分类（但 735 篇的 API 调用成本不低）
2. 增加更多细粒度的关键词规则
3. 提供一个手动重新分类的界面

目前的方案是"够用的自动化 + 可手动上传时修正"，这对一个个人知识库来说是合理的平衡。

### 关于"上传自动分类"的局限性

上传的文章存储在 `localStorage` 里。这意味着：
- 换设备/清缓存后数据会丢失
- 无法跨设备同步

如果要做持久化，可选方案包括：
- 用 GitHub Issues 或 Gist 当数据库（通过 GitHub API 读写）
- 接入 Supabase/Firebase 等 BaaS
- 生成新的 chunk JSON 文件并提交到仓库

### 关于性能

- 首屏加载：index.json (427KB) + HTML (52KB) ≈ ~500KB，gzip 后约 150-200KB，可接受
- 文章正文：最大的 chunk 是"2026年一月"（1.3MB），首次点击该分组内文章时会有短暂加载
- ECharts 库本身约 800KB（CDN 缓存后不再重复加载）

### 关于搜索降级

从全文搜索降级到标题+摘要搜索是一个明确的取舍。全文搜索需要所有正文在内存里，而我们选择了按需加载以优化首屏性能。两者不可兼得，除非构建额外的搜索索引。

### 最终架构总结

```
Excel 文件
    ↓ openpyxl 提取
735 篇原始文章 (cell → article)
    ↓ 标题提取 + 体裁分类 + 主题分类
结构化数据 (id, title, genre, topic, origin, preview, content)
    ↓ 索引/内容分离
index.json (元数据) + 21 × chunk_N.json (正文)
    ↓ 静态 HTML + fetch 按需加载
GitHub Pages 在线知识库
```

从一个混乱的 Excel 到一个有三维导航、6 张图表、分享和卡片导出功能的在线知识库——这个过程本身就是一次"从混沌到秩序"的认知炼金术。恰好，这也是 Refine无羡 的文章里一直在探讨的主题。
